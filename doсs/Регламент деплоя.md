# Регламент деплоя — Urban Running Games

## Подготовка к деплою

**Перед деплоем необходимо:**

1. **Проверить изменения:**
   - Убедиться, что все изменения закоммичены и запушены в репозиторий
   - Проверить, что тесты проходят (если есть)
   - Убедиться, что миграции БД готовы и протестированы локально

2. **Проверить окружение:**
   - Проверить наличие всех необходимых переменных окружения в `.env` на сервере:
     - `APP_ENV=production`
     - `APP_DEBUG=false`
     - `LOG_CHANNEL` и `LOG_LEVEL`
     - Настройки БД (`DB_*`)
     - Настройки почты (`MAIL_*`) для отправки уведомлений
   - Убедиться, что версии PHP (8.4), MySQL (8) соответствуют требованиям проекта
   - Проверить доступность сервисов (БД, почта, внешние API)
   - Убедиться, что права доступа к `storage/` и `bootstrap/cache/` настроены корректно
   - Для обработки изображений (галерея, превью): расширение PHP **GD** или **Imagick** (требуется для `intervention/image-laravel`)

3. **Резервное копирование:**
   - Создать бэкап базы данных: `mysqldump -u user -p database > backup_$(date +%Y%m%d_%H%M%S).sql`
   - Сохранить текущую версию кода (тег или бэкап директории)
   - Проверить наличие бэкапов медиафайлов (если используются)

## Процесс деплоя (Production)

**Шаг 1: Подключение к серверу**
```bash
ssh user@production-server
cd /var/www/urban-running-games
```

**Шаг 2: Переключение в режим обслуживания (опционально)**
```bash
php artisan down --message="Обновление системы" --retry=60
```

**Шаг 3: Получение последних изменений**
```bash
git fetch origin
git checkout master  # или main
git pull origin master
```

**Шаг 4: Установка зависимостей**
```bash
composer install --no-dev --optimize-autoloader
npm install --production
npm run build
```

**Шаг 5: Применение миграций БД**
```bash
php artisan migrate --force
```

**Примечание:** Если это первый деплой или были добавлены новые миграции, убедитесь, что все таблицы созданы корректно:
- `events`, `cities`, `partners`, `event_partners`
- `contact_messages`
- `media_assets` (в т.ч. колонка `thumbnail_path` для превью)
- `albums`, `album_items`

**Шаг 6: Очистка кэша**
```bash
php artisan config:clear
php artisan cache:clear
php artisan route:clear
php artisan view:clear
php artisan optimize
```

**Шаг 7: Проверка прав доступа**
```bash
chmod -R 755 storage bootstrap/cache
chown -R www-data:www-data storage bootstrap/cache
```

**Шаг 8: Выход из режима обслуживания**
```bash
php artisan up
```

**Шаг 9: Проверка работоспособности**
- Проверить главную страницу сайта (`/`)
- Проверить навигацию и все основные разделы:
  - О команде (`/about`)
  - Правила забега (`/rules`)
  - События (`/events`)
  - Архив событий (`/archive`)
  - **Фотогалерея** (`/gallery`) — список событий с альбомами, открытие альбома и отображение фото
  - Партнёры (`/partners`)
  - Контакты (`/contact`)
  - Рейтинг (`/rating`)
  - Юридические страницы (`/privacy`, `/consent`, `/terms`, `/returns`)
- Проверить форму обратной связи (отправка сообщения)
- Проверить отображение событий (предстоящие и архивные)
- Проверить карточки событий (детальная информация, блок «Фотогалерея» со ссылками на альбомы)
- Проверить загрузку изображений по URL `/storage/...` (фото и превью в галерее). При 404 на симлинке раздача идёт через Laravel (fallback).
- Проверить админ-панель: раздел «Альбомы», массовая загрузка фото в альбом, превью в списке
- Проверить логи на наличие ошибок: `tail -f storage/logs/laravel.log`

## Процесс деплоя (Staging)

Staging деплой выполняется аналогично Production, но:
- Используется отдельный vhost или поддомен (например, `stage.urban-running-games.ru`)
- Можно использовать `--dev` флаги для composer
- Можно оставить `APP_DEBUG=true` для отладки
- Миграции применяются с предупреждениями (без `--force`)

## Автоматизированный деплой через CI/CD

**GitHub Actions workflow** (`.github/workflows/deploy.yml`):

```yaml
name: Deploy to Production

on:
  push:
    branches:
      - master

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup PHP
        uses: shivammathur/setup-php@v2
        with:
          php-version: '8.4'
          
      - name: Deploy via SSH
        uses: appleboy/ssh-action@master
        with:
          host: ${{ secrets.PROD_HOST }}
          username: ${{ secrets.PROD_USER }}
          key: ${{ secrets.PROD_SSH_KEY }}
          script: |
            cd /var/www/urban-running-games
            git pull origin master
            composer install --no-dev --optimize-autoloader
            php artisan storage:link || true
            php artisan migrate --force
            php artisan config:clear
            php artisan cache:clear
            php artisan route:clear
            php artisan view:clear
            php artisan optimize
```

## Откат (Rollback)

**Если деплой прошёл неудачно:**

1. **Быстрый откат к предыдущей версии:**
```bash
git log --oneline  # найти предыдущий коммит
git checkout <previous-commit-hash>
composer install --no-dev --optimize-autoloader
php artisan migrate:rollback  # если были миграции
php artisan config:clear
php artisan cache:clear
php artisan optimize
php artisan up
```

2. **Восстановление БД из бэкапа:**
```bash
mysql -u user -p database < backup_YYYYMMDD_HHMMSS.sql
```

3. **Проверка после отката:**
- Убедиться, что сайт работает
- Проверить логи на ошибки
- Уведомить команду о проблеме

## Локальная разработка (встроенный PHP-сервер)

Для корректной раздачи файлов по `/storage/*` (фото галереи) запускайте сервер **с роутером**:
```bash
php -S localhost:8000 -t public public/router.php
```
Роутер передаёт запросы к `/storage/...` в Laravel; без него возможны 404 на медиа. Команда `php artisan serve` роутер не использует.

- Проверить отображение событий на главной странице и странице событий
- Проверить корректность работы архива событий
- Проверить загрузку фото на странице фотогалереи и в альбомах

**В течение 24 часов:**
- Проверять логи ежечасно
- Отслеживать пользовательские обращения
- Мониторить производительность БД
