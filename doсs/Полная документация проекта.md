- **Sentry (ошибки):** пока не требуется

- **1 VPS** (stage можно тем же сервером, отдельным vhost):
  - Nginx + PHP-FPM
  - MySQL
- **CI/CD:** GitHub Actions (или GitLab CI) → deploy по SSH (Envoy/Deployer/простой rsync+artisan)
- **Cron:** Laravel Scheduler (раз в минуту)
- **Регламент деплоя:** см. отдельный документ `doсs/Регламент деплоя.md`

### 3.9. Базовое логирование ошибок

✅ **Базовое логирование ошибок настроено:**

- **Laravel по умолчанию** логирует все исключения через встроенный механизм обработки ошибок
- Конфигурация логирования находится в `config/logging.php`
- По умолчанию используется канал `stack`, который пишет в `single` (файл `storage/logs/laravel.log`)
- Дополнительно настроен отдельный канал `errors` для структурированного логирования ошибок (файл `storage/logs/errors.log`)
- Ротация логов настроена через `daily` драйвер (14 дней для основного лога, 30 дней для лога ошибок)

**Проверка логирования:**
- Логи доступны в `storage/logs/laravel.log` и `storage/logs/errors.log`
- Уровень логирования настраивается через переменную окружения `LOG_LEVEL` (по умолчанию `debug`)
- В production рекомендуется установить `LOG_LEVEL=error` или `LOG_LEVEL=warning`

## 4. Структура базы данных и миграции

### 4.0. Важные правила, которые закладываем в схему

> **Примечание:** Данный раздел описывает полную схему БД из исходного документа. Однако, согласно ТЗ, **не реализуются:** продажа слотов участия, личные кабинеты, рейтинги и команды. Поэтому в реальных миграциях проекта используются только те таблицы, которые необходимы для реализации функционала по ТЗ.

1. 1 слот = 1 команда, команда создаётся только после успешной оплаты.
2. Мягкий резерв слота на 15 минут на время оплаты.
3. Команда = 4 человека (капитан + 3), вступление по приглашению, только после авторизации.
4. Дедлайн вступления: за 72 часа до старта (лучше хранить как поле join_deadline_at, чтобы "зафиксировать" на момент публикации события).
5. Ограничение "1 команда на пользователя" в рамках события.
6. Промокоды: фикс/процент + ограничения (период, лимиты, список событий, мин. сумма).
7. Мерч: каталог + варианты + "заказ по заявке" + статусы + доставка (500₽/1000₽/самовывоз).
8. Платёжный провайдер — Т-банк (вебхуки и сверка оплат).

#### team_members
- id BIGINT PK
- team_id BIGINT FK → teams.id
- event_id BIGINT FK → events.id ✅ (денормализация)

- role ENUM('captain','member')
- status ENUM('active','removed') DEFAULT 'active'
- joined_at DATETIME
- created_at

**Индексы/ограничения:**

- UNIQUE (team_id, user_id) — один и тот же пользователь не добавится дважды в одну команду
- UNIQUE active_event_user_key — не даст пользователю иметь две активные записи в одном событии
- INDEX (event_id, user_id) — быстрые проверки/выборки

**Как сделать active_event_user_key в MySQL:**

Добавляем вычисляемую STORED-колонку:
```
active_event_user_key = CASE WHEN status='active' THEN CONCAT(event_id, ':', user_id) ELSE NULL END
```
и делаем по ней UNIQUE индекс.

MySQL допускает много NULL в UNIQUE — поэтому у "removed" ограничение не мешает, а для "active" — строго один вариант.

### 4.14. Критичная транзакция: резерв + оплата + создание команды (как должно работать)

**При переходе к оплате:**

- SELECT ... FOR UPDATE по events
- считаем available = slots_total - slots_sold - active_reservations
- создаём slot_reservations(reserved_until = now()+15min)
- создаём participation_orders(status=reserved)

**По вебхуку "paid":**

- транзакция
- проверяем, что резерв ещё валиден/или слот ещё есть (критические проверки лимитов)
- ставим participation_orders.status=paid, payments.status=paid
- инкремент events.slots_sold
- создаём teams + team_invites + team_members(captain)
